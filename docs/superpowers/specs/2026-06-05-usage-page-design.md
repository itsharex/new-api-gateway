# 用量页改造设计

- 日期：2026-06-05
- 范围：管理后台 `用量` 菜单页
- 状态：已完成设计确认，待实现规划

## 背景

当前 `用量` 页存在两个明显问题：

1. 页面默认必须先输入员工用户名，导致在没有明确目标员工时，菜单第一页几乎没有可用信息。
2. 员工趋势图在 `1d / 7d / 30d` 切换后只展示有数据的时间点，缺失时间桶不会补齐，容易出现“30d 只显示昨天到今天横轴”的误解。

同时，`概览` 页已经展示最近 30 天的全局 token 趋势，因此 `用量` 页不应简单复制同一张趋势图，而应承担“全局用量分析 + 当前页内钻取员工详情”的职责。

## 目标

1. 让 `用量` 页在默认进入时就有分析价值，不再依赖先输入员工。
2. 默认以最近 30 天的全局用量分析为主，突出员工排行与模型分布。
3. 保留常驻员工搜索框，并支持从搜索或榜单点击进入员工详情。
4. 员工详情必须在当前页展开，不跳转到新页面。
5. 修正趋势图时间轴行为：
   `1d` 按小时展示最近 24 小时并补齐完整 24 个小时桶；
   `7d / 30d` 按天展示并补齐完整天数桶；
   无数据桶使用 `0` 填充。

## 非目标

1. 不改造 `概览` 页的信息结构。
2. 不新增部门、组织树等更大范围的筛选体系。
3. 不在本次设计中引入新的跨页详情路由。
4. 不在本次设计中扩展导出、打印、批量对比等高级分析能力。

## 页面结构

`用量` 页改为单页分析面板，分成“全局层”和“员工详情层”。

### 全局层

全局层默认始终可见，固定展示最近 30 天口径的数据，包含：

1. 常驻搜索框
   - 位于页面上方显著位置。
   - 用于直接搜索并选中员工。
   - 默认提供轻量候选建议，至少展示最近活跃员工或关键词匹配建议中的一种。

2. 全局摘要卡片
   - 只保留少量高价值指标，避免与 `概览` 重复。
   - 固定包括：最近 30 天总 Token、活跃员工数、请求数、活跃模型数。

3. `Top 员工榜`
   - 作为页面主区块。
   - 默认按最近 30 天 `Total Tokens` 降序排列。
   - 每一行可点击，点击后在当前页展开该员工详情。

4. 模型分布 / Top Models
   - 作为次级全局区块。
   - 展示最近 30 天模型总 Token 分布或 Top Models 排名。
   - 目的不是重做概览趋势，而是补充“用量由哪些模型构成”。

### 员工详情层

员工详情层位于全局层下方，默认折叠，仅在选中员工后展开。

内容顺序固定为：

1. 当前查看员工标题与收起动作
2. 员工汇总卡片
3. 员工趋势图
4. 员工 Model 汇总表

这样可以先给量级，再给走势，最后给组成。

## 交互设计

### 员工选择

1. 搜索框和员工榜是并列入口。
2. 搜索选中员工与点击员工榜，最终都落到同一个 `selectedEmployee` 状态。
3. 页面同一时刻只展开一个员工详情。
4. 当用户点击另一个员工时，详情区直接切换到新员工，不叠加多个详情区。

### 当前页展开

1. 选中员工后，不跳页，不覆盖全局层。
2. 全局榜单、模型分布、搜索框继续保留在详情区上方。
3. 详情区提供明确的“收起详情”动作，回到仅看全局层的状态。

### 趋势图规则

1. `1d`
   - 表示最近 24 小时。
   - 按小时展示。
   - 横轴固定补齐 24 个小时桶。
   - 无数据小时补 `0`。

2. `7d`
   - 表示最近 7 天。
   - 按天展示。
   - 横轴固定补齐 7 个自然日桶。
   - 无数据日期补 `0`。

3. `30d`
   - 表示最近 30 天。
   - 按天展示。
   - 横轴固定补齐 30 个自然日桶。
   - 无数据日期补 `0`。

4. 补充提示
   - 即使图上补齐了完整时间轴，也要在详情区提示“当前范围内仅 N 个时间桶有实际流量”。
   - 提示仅作辅助，不改变横轴长度。

## 前后端契约调整

### 后端接口

继续沿用 `/admin/api/usage` 作为主接口，但响应需拆成两层语义：

1. `global_usage`
   - 默认进入 `用量` 页时返回。
   - 固定最近 30 天口径。
   - 包含全局摘要卡片、Top 员工榜、模型分布 / Top Models 所需数据。

2. `employee_usage`
   - 仅在传入 `username` 时返回。
   - 包含当前员工详情、趋势图、Model 汇总表所需数据。

现有 `usage` 聚合结果如被其他区域复用可继续保留，但 `用量` 页前端必须改为优先消费 `global_usage` 与 `employee_usage`，不再以“没有 username 就渲染空态”为主逻辑。

### 员工趋势数据生成

后端负责按范围选择聚合粒度并补齐时间轴，不把这部分语义丢给前端：

1. `1d` 查询 `usage_aggregates` 的 `hour` bucket。
2. `7d / 30d` 查询 `usage_aggregates` 的 `day` bucket。
3. 返回前完成完整时间桶填充。
4. 响应中的时间点应按时间顺序输出，便于图表直接渲染。

这样前端拿到的就是“已成型的展示数据”，不需要再自己推断缺失日期或小时。

## 前端状态设计

当前 `state.usage` 仅有 `username / range / model`，需要扩展为以页面状态为中心的结构，至少包含：

1. `selectedEmployee`
   - 当前详情区正在查看的员工。

2. `range`
   - 当前详情区趋势图范围，支持 `1d / 7d / 30d`。

3. `model`
   - 当前员工详情的模型筛选值。

4. `globalLoaded`
   - 标记全局层是否已加载完成。

搜索和榜单点击都只更新 `selectedEmployee`，然后重新请求或重绘详情区；不应重置上方全局层。

## 空态与错误处理

1. 全局层加载失败
   - 整页显示错误，因为这是 `用量` 页的默认主体。

2. 员工详情加载失败
   - 仅在详情区显示错误。
   - 全局层保持可用。

3. 员工无数据
   - 详情区仍显示完整时间轴和 0 值图。
   - 同时提示“该时间范围内暂无用量”。

4. 初始未选中员工
   - 不再显示“必须先输入员工”的空白页。
   - 默认展示完整全局层。

## 测试策略

### 后端测试

1. 默认进入 `用量` 页时返回 `global_usage`。
2. 传入 `username` 时同时返回 `employee_usage`。
3. `1d` 返回固定 24 个小时点。
4. `7d` 返回固定 7 个天点。
5. `30d` 返回固定 30 个天点。
6. 缺失时间桶补 `0`。
7. `1d` 使用 `hour` bucket，`7d / 30d` 使用 `day` bucket。

### 前端测试

1. 默认进入 `用量` 页直接渲染全局层，不再空白。
2. 搜索员工后在当前页展开详情区。
3. 点击员工榜后在当前页展开详情区。
4. 切换到其他员工时详情区单实例切换，不叠加。
5. 切换 `1d / 7d / 30d` 后横轴长度稳定。
6. 详情区加载失败时不影响上方全局层。

## 实现顺序建议

1. 先补后端 `global_usage` 与员工趋势完整时间轴能力。
2. 再调整前端状态与默认首屏渲染。
3. 最后补员工榜点击展开、搜索联动和详情区交互。

这样可以先把数据契约稳定下来，再让前端围绕稳定结构改版，降低联调反复成本。
